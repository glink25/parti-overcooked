# 地图设计审计与重构说明

## 设计基准

《胡闹厨房》的关卡并不是先堆叠机关再填工位，而是先确定一段需要玩家沟通的“协作故事”。Team17 对《胡闹厨房 2》的回顾把这种方法概括为 gameplay first：每个厨房先确定一个核心点子，例如滑动台面、传送门、分隔厨师或传送带，再让关卡形成自己的节奏。Ghost Town Games 也强调从期望玩家如何行动出发，反复调整分工、工位和 1–4 人组合。

- [Team17：Overcooked 2 关卡设计回顾](https://www.team17.com/news/team17s-100-games-the-final-chapter-2018-overcooked-2-planet-alpha-yokus-island-express-more)
- [Ghost Town Games：关卡从玩家故事与分工开始](https://www.pushsquare.com/news/2018/08/interview_chewing_the_fat_with_overcooked_2_developer_ghost_town_games)

本项目据此采用四项地图合同：机关必须改变路线或分工；每个玩家分区持续有事可做；危险必须提前可读；动态变化不得让整局进入不可恢复的硬锁。

## 城堡门阵协议

城门布局定义包含 `groups` 和 `presets`。每个 group 声明名称、方向及三个碰撞格；每个 preset 声明同时开放的两个门组。运行时状态公开：

```text
activePresetId / nextPresetId / remaining / warning
gates[]: id / label / cells / open / willOpen / willClose
```

洗牌袋保证四个门阵在一轮内各出现一次，跨轮首项不得与当前门阵重复。预警开始时便确定 `nextPresetId`，因此占位延迟不会重抽路线。服务端碰撞、投掷阻挡和客户端预测都使用同一组 `cells`。

## 封闭货运与复活协议

所有传送带路径只能使用水平、垂直线段和 90° 转角。带体与普通台面等高，服务端权威碰撞和客户端预测使用相同的 0.8 格宽碰撞矩形；玩家不能进入带体，带上物品也不能被直接拾取或投放。

`conveyorPort` 通过 `conveyorId / portMode / pathPoint` 连接货运线。输入台允许放入或取回等待货物，空位出现时自动装载；输出台只允许取货。非循环线在输出被占用时排队，环形线则继续循环等待任一输出空闲。斜线路径在地图加载阶段直接报错。

玩家每轮记录 `roundSpawnSlot`。坠落复活、同轮断线重连和移动平台坐标换算均以该槽为准，不再根据坠落位置选择最近检查点。环岛因此始终保持外环玩家回外环、中央厨师回中央，不会因集体坠落而聚集到同一岛屿。

## 验收指标

- 经典、太空、城堡没有任何安全地板直接邻接无墙虚空。
- 环岛和雪山只有显式装卸口或裂谷保留安全地板到危险地形的开放边。
- 全部传送带均具有至少一个输入台和输出台，不含斜线段；出生点与检查点不得落在工位或带体内部。
- 城堡四种门阵下东西南北各翼始终可经外围环道互达，且东西门开放时的东西最短路短于南北门阵。
- 两人和四人使用同一布局与机关节奏；人数差异继续由订单生成频率调整。
- 地图涉及的每个固定工位至少保留一个可站立交互面，菜谱池需要的全部食材箱均存在。
